§ I

Agent Efficiency Foundations

01 Reproducible Development Environment weight 6 · Note ▸
Why it matters

Without a clean-checkout install path, agents waste cycles guessing at undocumented local setup and hit “works on my machine” failures.

What good looks like

A fresh clone can be installed, run, tested, and built using documented commands tied to real files — devcontainer, Dockerfile, Makefile, justfile, or a setup section in CLAUDE.md.

Observed
  • README has setup steps but no Makefile or devcontainer.
  • package.json scripts present but undocumented in CLAUDE.md.
Remediation
  • Add a justfile with install / dev / test / build targets.
02 Package Manager & Runtime Pinning weight 4 · Ok ▸
Why it matters

When the package manager or runtime version is ambiguous, agents may run npm in a pnpm project, use the wrong runtime, or generate a second lockfile that conflicts with CI.

What good looks like

One package manager per ecosystem with a single lockfile; runtime versions pinned via .nvmrc / .tool-versions / pyproject.toml.

Observed
  • pnpm-lock.yaml present; packageManager: pnpm@9.7.0.
  • .nvmrc pins node 22.6.0.
03 Repo Map weight 5 · Ko ▸
Why it matters

Without a top-level map, agents waste tokens grepping the whole tree to figure out where features live — particularly in monorepos.

What good looks like

A REPO_MAP.md lists every top-level folder with a one-line purpose, ideally with owner and main entry point.

Observed
  • No REPO_MAP.md found.
  • README lists 47 packages — none with descriptions.
Remediation
  • Create REPO_MAP.md with one-line purpose for each top-level folder.
04 Context Cascade weight 4 · Ok ▸
Why it matters

A single bloated CLAUDE.md forces agents to load thousands of lines of mostly-irrelevant guidance on every task, burning context.

What good looks like

The root CLAUDE.md stays focused on global rules (under ~200 lines); subtree files carry local commands and gotchas where they apply.

Observed
  • Root CLAUDE.md is 142 lines.
  • Three subtree files in packages/api, packages/web, packages/cli.
05 CLAUDE.md Quality weight 10 · Note ▸
Why it matters

A CLAUDE.md full of generic platitudes or stale instructions costs context without adding value, and sometimes actively misleads.

What good looks like

Concise, repo-specific, aligned with actual scripts and CI. Captures the non-obvious; skips standard language conventions.

Observed
  • 22 lines of generic “write clean code” advice.
  • Verification commands present and match package.json scripts.
Remediation
  • Remove generic style advice; keep only repo-specific gotchas.
§ II

Context & Tooling

06 Noise Filter weight 6 · Ko ▸
Why it matters

When node_modules, dist, vendor, coverage, and generated SDK folders are searchable, every grep returns hundreds of irrelevant hits.

What good looks like

A committed .claude/settings.json uses permissions.deny to block reads of noisy/generated paths that actually exist in the repo.

Observed
  • No .claude/settings.json found.
  • node_modules and dist are searchable.
Remediation
  • Add .claude/settings.json with permissions.deny for node_modules/, dist/, .next/.
07 Symbol Lookup weight 8 · Ok ▸
Why it matters

Without LSP-backed code intelligence, agents fall back to grep for definitions and references — slow, noisy, and easily fooled by name collisions.

What good looks like

An MCP server with LSP support (e.g. serena) is configured for the dominant language(s) and visible in .mcp.json.

Observed
  • .mcp.json declares serena for TypeScript and Python.
08 Large-Repo File Discovery weight 5 · Note ▸
Why it matters

In a monorepo with thousands of files, blind grep and find produce slow, noisy results and exhaust context before the agent finds the right file.

What good looks like

At least one of: REPO_MAP.md, a workspace manifest, a fileSuggestion config, or a documented “how to find files” section in CLAUDE.md.

Observed
  • pnpm-workspace.yaml present.
  • No REPO_MAP.md; tree is < 300 files.
Remediation
  • Add a short “where to find things” section in CLAUDE.md.
§ III

Verification & Safety

09 Deterministic Checks weight 10 · Ko ▸
Why it matters

When quality enforcement is “remember to run the linter” in prose, agents and humans forget. Hooks make checks unavoidable.

What good looks like

.claude/settings.json wires PostToolUse hooks on Edit/Write/MultiEdit to real project commands.

Observed
  • No hooks declared in .claude/settings.json.
  • lint and typecheck scripts exist but are not wired to PostToolUse.
Remediation
  • Wire PostToolUse hook to pnpm lint --filter on Edit/Write.
10 Verifiable Work Loop weight 9 · Ok ▸
Why it matters

An agent that cannot verify its own work has to hand back to a human just to know whether the change compiles.

What good looks like

CLAUDE.md has an explicit “Verification” section listing the exact install / lint / typecheck / test / build commands.

Observed
  • CLAUDE.md has explicit Verification section with install/lint/typecheck/test/build commands.
11 CI Parity weight 4 · Note ▸
Why it matters

When local agent checks are weaker than CI, work passes locally and fails in the pipeline, forcing rework.

What good looks like

Every meaningful CI check has a documented local equivalent, ideally wrapped in a single make verify.

Observed
  • CI runs pnpm verify which is documented locally.
  • ESLint config differs slightly between local and CI.
Remediation
  • Align local ESLint with .github/workflows/ci.yml.
12 Fast Feedback / Focused Tests weight 6 · Ok ▸
Why it matters

If the only documented way to test is “run the full suite”, agents either wait minutes per iteration or skip testing entirely.

What good looks like

CLAUDE.md documents how to run a single test file, a package/module scope, and the full suite.

Observed
  • Single-file test command documented: pnpm test -- path/file.
  • Watch mode documented.
13 Permission Hardening weight 6 · Ko ▸
Why it matters

Without committed deny rules, a coding agent can force-push, hard-reset, rm -rf, or edit .env and key files.

What good looks like

A committed .claude/settings.json has a permissions.deny block covering destructive bash and sensitive files.

Observed
  • .claude/settings.json has no deny block.
  • rm -rf and git push --force are not blocked.
Remediation
  • Add deny entries for destructive bash and sensitive file reads.
14 Local Config Hygiene weight 3 · Ok ▸
Why it matters

When CLAUDE.local.md, settings.local.json, .env files, or private keys are not gitignored, personal config and secrets can be committed by accident.

What good looks like

.gitignore excludes Claude-local files, .env variants (with .env.example whitelisted), and key/credential patterns.

Observed
  • .gitignore covers CLAUDE.local.md, settings.local.json, .env*.
  • No local files tracked.
§ IV

Reusable Agent Workflows

15 Just-in-Time Skill weight 3 · Ok ▸
Why it matters

Stuffing every workflow into CLAUDE.md means agents load it all every session, wasting context.

What good looks like

.claude/skills/ contains narrow, well-described skills with a clear “when to use” description.

Observed
  • .claude/skills/release/ and .claude/skills/migration/ present with clear descriptions.
16 Agent Output / PR Hygiene weight 3 · Note ▸
Why it matters

Without explicit PR expectations, agents improvise summaries that omit tests run, risks, rollback notes, or screenshots.

What good looks like

A PR template tells agents to include summary, verification commands, risks, rollback notes, and UI screenshots.

Observed
  • .github/PULL_REQUEST_TEMPLATE.md present but no risk/rollback section.
Remediation
  • Add Risks and Rollback sections to the PR template.
17 Custom Subagents weight 4 · Ko ▸
Why it matters

Generic subagents handle generic work, but recurring high-risk reviews (security, migrations, performance) benefit from a focused agent.

What good looks like

.claude/agents/ contains project-specific subagents matched to the repo’s actual risk surface.

Observed
  • No .claude/agents/ folder.
  • Repo handles auth and migrations but uses generic agents.
Remediation
  • Create .claude/agents/security-reviewer.md and migration-reviewer.md.
